# RPC — বাংলা ব্যাখ্যা

_টপিক নম্বর: 077_

## গল্পে বুঝি

মন্টু মিয়াঁ internal service-to-service communication-এ typed contract, fast serialization, এবং strong tooling চান।

`RPC` টপিকটা RPC-style communication বোঝায় যেখানে remote call-কে local function call-এর মতো model করা হয়, কিন্তু network failure reality ভুললে সমস্যা হয়।

Typed schemas/protobuf/IDL approach development speed বাড়াতে পারে, তবে compatibility, retries, timeouts, and observability design করতে হয়।

ইন্টারভিউতে RPC/gRPC বললে synchronous coupling ও failure propagation কিভাবে handle করবেন সেটা বললে maturity বোঝায়।

সহজ করে বললে `RPC` টপিকটি নিয়ে সোর্স নোটের মূল কথাটা হলো: RPC (Remote Procedure Call) is a style where a program calls a remote function/service as if calling a local procedure।

বাস্তব উদাহরণ ভাবতে চাইলে `Google`-এর মতো সিস্টেমে `RPC`-এর trade-off খুব স্পষ্ট দেখা যায়।

---

### `RPC` আসলে কীভাবে সাহায্য করে?

`RPC` ব্যবহার করার আসল মূল্য হলো requirement, behavior, এবং trade-off-কে একইসাথে পরিষ্কার করে design decision নেওয়া।

- communication style/protocol choice-কে workload requirement (latency, compatibility, reliability) এর সাথে map করতে সাহায্য করে।
- timeout, retry, idempotency, versioning, আর error semantics design discussion-এ আনতে সাহায্য করে।
- public API বনাম internal service call-এর protocol choice আলাদা করার যুক্তি দেয়।
- debugging/tooling/observability impact explain করলে protocol decision আরও বাস্তবসম্মত হয়।

---

### কখন `RPC` বেছে নেওয়া সঠিক?

মন্টু নিজের কাছে কয়েকটা প্রশ্ন করে:

- কোথায়/কখন use করবেন? → Internal microservices সাথে strong contracts, typed ক্লায়েন্টগুলো, এবং পারফরম্যান্স needs.
- Business value কোথায় বেশি? → এটি simplifies সার্ভিস-টু-সার্ভিস integration এবং hides transport details behind typed interfaces.
- public API আর internal service call-এ একই protocol লাগবে নাকি আলাদা?
- timeout, retry, idempotency, versioning কীভাবে handle করবেন?

এই প্রশ্নগুলোর উত্তরে topicটা product requirement-এর সাথে fit করলে সেটাই সঠিক choice।

---

### কিন্তু কোথায় বিপদ?

এই টপিক ভুলভাবে ব্যবহার করলে সাধারণত এই সমস্যা দেখা দেয়:

- ভুল context: Public APIs requiring broad interoperability এবং browser-friendly semantics.
- ইন্টারভিউ রেড ফ্ল্যাগ: Treating RPC as automatically bad কারণ it হলো না REST.
- Hiding network ফেইলিউর realities behind local-call mental models.
- Ignoring deadline propagation এবং রিট্রাইগুলো.
- Poor contract versioning যা breaks ক্লায়েন্টগুলো.

তাই মন্টু এক জিনিস পরিষ্কার রাখে:

> `RPC` শুধু term না; context + trade-off + user impact একসাথে define না করলে design answer অসম্পূর্ণ।

## এক লাইনে

- `RPC` client/service communication protocol বা API style বাছাইয়ে semantics, latency, reliability, ও compatibility trade-off বোঝায়।
- এই টপিকে বারবার আসতে পারে: protocol semantics, timeout/retry, versioning, idempotency, observability

## এটা কী (থিওরি)

সহজ ভাষায় সংজ্ঞা ও মূল ধারণা:

- বাংলা সারাংশ: `RPC` services/clients-এর communication contract, semantics, এবং network behavior বোঝার টপিক।

- RPC (Remote Procedure Call) হলো a style যেখানে a program calls a remote function/সার্ভিস as যদি calling a local procedure.

## কেন দরকার

কেন এই ধারণা/প্যাটার্ন দরকার হয়:

- বাংলা সারাংশ: একই communication style সব workload-এ fit করে না; latency, compatibility, reliability অনুযায়ী protocol/API design দরকার।

- এটি simplifies সার্ভিস-টু-সার্ভিস integration এবং hides transport details behind typed interfaces.

## কীভাবে কাজ করে (সিনিয়র-লেভেল ইনসাইট)

বাস্তবে/প্রোডাকশনে সাধারণত এভাবে কাজ করে:

- বাংলা সারাংশ: timeout, retry, idempotency, versioning, error semantics, এবং observability plan ছাড়া protocol discussion অসম্পূর্ণ থাকে।

- RPC frameworks define সার্ভিস contracts, serialization, রিট্রাইগুলো, এবং transport handling.
- এটি হলো efficient জন্য internal সিস্টেমগুলো but পারে create tight coupling যদি versioning এবং backward compatibility হলো weak.
- Compare সাথে REST: RPC emphasizes actions/methods; REST emphasizes resources এবং uniform interface semantics.

## বাস্তব উদাহরণ

একটি পরিচিত প্রোডাক্ট/সিস্টেমের উদাহরণ:

- বাংলা সারাংশ: বাস্তব উদাহরণে খেয়াল করুন, `RPC` একই product-এর ভিন্ন feature/path-এ ভিন্নভাবে apply হতে পারে; context-টাই আসল।

- **Google** internal সিস্টেমগুলো have long used RPC-style communication জন্য typed, performant সার্ভিস interactions.

## ইন্টারভিউ পার্সপেক্টিভ

ইন্টারভিউতে উত্তর দেওয়ার সময় যেসব দিক বললে ভালো হয়:

- বাংলা সারাংশ: ইন্টারভিউতে `RPC` explain করার সময় scope, user impact, trade-off, failure case, আর “কখন ব্যবহার করবেন না” — এই পাঁচটি দিক বললে উত্তর শক্তিশালী হয়।

- কখন ব্যবহার করবেন: Internal microservices সাথে strong contracts, typed ক্লায়েন্টগুলো, এবং পারফরম্যান্স needs.
- কখন ব্যবহার করবেন না: Public APIs requiring broad interoperability এবং browser-friendly semantics.
- একটা কমন ইন্টারভিউ প্রশ্ন: \"What হলো the ট্রেড-অফ মাঝে RPC এবং REST জন্য internal সার্ভিসগুলো?\"
- রেড ফ্ল্যাগ: Treating RPC as automatically bad কারণ it হলো না REST.

## কমন ভুল / ভুল ধারণা

যে ভুলগুলো অনেকেই করে:

- বাংলা সারাংশ: `RPC`-এ সাধারণ ভুল হলো শুধু term/definition বলা; context, limitation, operational cost, এবং user-visible impact না বলা।

- Hiding network ফেইলিউর realities behind local-call mental models.
- Ignoring deadline propagation এবং রিট্রাইগুলো.
- Poor contract versioning যা breaks ক্লায়েন্টগুলো.

## দ্রুত মনে রাখুন

- রেড ফ্ল্যাগ মনে রাখুন: Treating RPC as automatically bad কারণ it হলো না REST.
- কমন ভুল এড়ান: Hiding network ফেইলিউর realities behind local-call mental models.
- Routing/communication টপিকে latency, retry behavior, এবং observability উল্লেখ করুন।
- কেন দরকার (শর্ট নোট): এটি simplifies সার্ভিস-টু-সার্ভিস integration এবং hides transport details behind typed interfaces.
