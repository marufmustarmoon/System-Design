# TCP — বাংলা ব্যাখ্যা

_টপিক নম্বর: 074_

## গল্পে বুঝি

মন্টু মিয়াঁ `TCP` নিয়ে ভাবছেন কারণ communication requirement সবসময় একই না: কোথাও reliability দরকার, কোথাও low-latency + low-overhead বেশি গুরুত্বপূর্ণ।

`TCP` টপিকটা transport-level trade-off বোঝায় - connection behavior, delivery characteristics, ordering, retransmission, congestion impact ইত্যাদি।

Protocol নির্বাচন feature-driven হওয়া উচিত: realtime stream, telemetry, request/response, control plane - প্রতিটা workload-এ priorities আলাদা।

ইন্টারভিউতে transport বললে উপরের application protocol/serialization layer-এর সাথে সম্পর্কও explain করা দরকার।

সহজ করে বললে `TCP` টপিকটি নিয়ে সোর্স নোটের মূল কথাটা হলো: TCP is a transport protocol that provides reliable, ordered byte-stream delivery between endpoints।

বাস্তব উদাহরণ ভাবতে চাইলে `Netflix`-এর মতো সিস্টেমে `TCP`-এর trade-off খুব স্পষ্ট দেখা যায়।

---

### `TCP` আসলে কীভাবে সাহায্য করে?

`TCP` ব্যবহার করার আসল মূল্য হলো requirement, behavior, এবং trade-off-কে একইসাথে পরিষ্কার করে design decision নেওয়া।

- communication style/protocol choice-কে workload requirement (latency, compatibility, reliability) এর সাথে map করতে সাহায্য করে।
- timeout, retry, idempotency, versioning, আর error semantics design discussion-এ আনতে সাহায্য করে।
- public API বনাম internal service call-এর protocol choice আলাদা করার যুক্তি দেয়।
- debugging/tooling/observability impact explain করলে protocol decision আরও বাস্তবসম্মত হয়।

---

### কখন `TCP` বেছে নেওয়া সঠিক?

মন্টু নিজের কাছে কয়েকটা প্রশ্ন করে:

- কোথায়/কখন use করবেন? → Most APIs, ডাটাবেজগুলো, এবং সার্ভিস-টু-সার্ভিস calls needing reliable delivery.
- Business value কোথায় বেশি? → Many applications need ডেটা to arrive in order এবং ছাড়া loss, সাথে retransmissions handled by the network stack.
- public API আর internal service call-এ একই protocol লাগবে নাকি আলাদা?
- timeout, retry, idempotency, versioning কীভাবে handle করবেন?

এই প্রশ্নগুলোর উত্তরে topicটা product requirement-এর সাথে fit করলে সেটাই সঠিক choice।

---

### কিন্তু কোথায় বিপদ?

এই টপিক ভুলভাবে ব্যবহার করলে সাধারণত এই সমস্যা দেখা দেয়:

- ভুল context: রিয়েল-টাইম voice/video/gaming scenarios যা পারে tolerate some loss but need lower ল্যাটেন্সি.
- ইন্টারভিউ রেড ফ্ল্যাগ: Describing TCP as মেসেজ-based এর বদলে a byte stream.
- Assuming one `send` maps to one `recv`.
- Ignoring connection management এবং timeouts.
- Blaming TCP জন্য application-layer inefficiency ছাড়া profiling.

তাই মন্টু এক জিনিস পরিষ্কার রাখে:

> `TCP` শুধু term না; context + trade-off + user impact একসাথে define না করলে design answer অসম্পূর্ণ।

## এক লাইনে

- `TCP` client/service communication protocol বা API style বাছাইয়ে semantics, latency, reliability, ও compatibility trade-off বোঝায়।
- এই টপিকে বারবার আসতে পারে: ordered delivery, retransmission, connection-oriented, latency trade-off, reliability

## এটা কী (থিওরি)

সহজ ভাষায় সংজ্ঞা ও মূল ধারণা:

- বাংলা সারাংশ: `TCP` services/clients-এর communication contract, semantics, এবং network behavior বোঝার টপিক।

- TCP হলো a transport protocol যা provides reliable, ordered byte-stream delivery মাঝে endpoints.

## কেন দরকার

কেন এই ধারণা/প্যাটার্ন দরকার হয়:

- বাংলা সারাংশ: একই communication style সব workload-এ fit করে না; latency, compatibility, reliability অনুযায়ী protocol/API design দরকার।

- Many applications need ডেটা to arrive in order এবং ছাড়া loss, সাথে retransmissions handled by the network stack.

## কীভাবে কাজ করে (সিনিয়র-লেভেল ইনসাইট)

বাস্তবে/প্রোডাকশনে সাধারণত এভাবে কাজ করে:

- বাংলা সারাংশ: timeout, retry, idempotency, versioning, error semantics, এবং observability plan ছাড়া protocol discussion অসম্পূর্ণ থাকে।

- TCP ম্যানেজ করে connection setup, flow control, congestion control, retransmission, এবং ordered delivery.
- Reliability simplifies application logic but পারে add ল্যাটেন্সি (handshakes, retransmission, head-of-line effects).
- Compare সাথে UDP: TCP হলো easier জন্য correctness এবং general apps, UDP হলো better জন্য ল্যাটেন্সি-tolerant loss-sensitive রিয়েল-টাইম ট্রাফিক in some cases.

## বাস্তব উদাহরণ

একটি পরিচিত প্রোডাক্ট/সিস্টেমের উদাহরণ:

- বাংলা সারাংশ: বাস্তব উদাহরণে খেয়াল করুন, `TCP` একই product-এর ভিন্ন feature/path-এ ভিন্নভাবে apply হতে পারে; context-টাই আসল।

- **Netflix** control APIs এবং most web ট্রাফিক rely on TCP-based protocols কারণ correctness এবং compatibility matter.

## ইন্টারভিউ পার্সপেক্টিভ

ইন্টারভিউতে উত্তর দেওয়ার সময় যেসব দিক বললে ভালো হয়:

- বাংলা সারাংশ: ইন্টারভিউতে `TCP` explain করার সময় scope, user impact, trade-off, failure case, আর “কখন ব্যবহার করবেন না” — এই পাঁচটি দিক বললে উত্তর শক্তিশালী হয়।

- কখন ব্যবহার করবেন: Most APIs, ডাটাবেজগুলো, এবং সার্ভিস-টু-সার্ভিস calls needing reliable delivery.
- কখন ব্যবহার করবেন না: রিয়েল-টাইম voice/video/gaming scenarios যা পারে tolerate some loss but need lower ল্যাটেন্সি.
- একটা কমন ইন্টারভিউ প্রশ্ন: \"Why choose TCP এর বদলে UDP জন্য এটি সার্ভিস?\"
- রেড ফ্ল্যাগ: Describing TCP as মেসেজ-based এর বদলে a byte stream.
